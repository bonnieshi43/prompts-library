## 三、按分类详述（类内：P0 → P1 → P2）

### A. 契约与可见行为

#### T1（P0）— UI 承诺与可执行契约

**何时用**：可见属性/文案/注释承诺了行为，某路径可能未落实。

**关键问**：代码是否在**每一条路径**上都落实了 UI 承诺？（与主提示词 **阶段 A / A1** 一致。）

**契约信号表**（扫描时对照）：

| 信号 | 示例 | 隐含规则 |
|------|------|----------|
| `<input accept="...">` | `accept=".pdf,.docx"` | 仅这些格式应被接受——**所有路径**（含拖拽）须一致 |
| 按钮 `disabled` | `disabled={!isFormValid()}` | 守卫须严密；需考虑**各种触发方式下是否仍可绕过** |
| 随状态变化的文案 | `{file ? '已选：…' : '拖拽文件到此处'}` | 所有设置 `file` 的路径须能更新该文案 |
| `required` 等 | `<TextField required>` | 空字段时须阻止提交 |
| 注释如「// 重置表单」 | `setFile(null); setFormData(...)` | **各层状态**都须重置，而非仅 React state |

#### M7（P1）— 加载 / 错误 / 空态契约

**何时用**：存在请求、提交、列表为空等三态。

**检查要点**：loading 时提交/删除是否应禁用；错误时文案与可重试路径；空态与 **T1** 文案是否一致。

---

### B. 交互路径

#### T2（P0）— 同目标多路径对称

**何时用**：同一用户目标有多条路径（鼠标 / **键盘** / **拖拽** / **父组件 ref 调用** 等）。

**路径枚举示例**：

```text
目标：选择文件
  路径 A → <input type="file"> onChange
  路径 B → 拖拽（onDrop）

目标：提交表单
  路径 A → 点击上传按钮
  路径 B → 键盘 Enter / 程序化调用（若存在）
```

**关键问**：**所有路径是否走同一套校验与业务逻辑？** 若一路绕过另一路的检查 → **高度可疑缺陷**。

---

### C. 状态与重置

#### T3（P0）— 全栈状态与重置

**何时用**：重置、清空、恢复、重开对话框等。

| 层级 | 示例 | 仅靠 React setState 能否复位？ |
|------|------|-------------------------------|
| React | `useState`、`useReducer` | 能 |
| DOM | `<input type="file">.value`、**滚动位置** | 否——常需 ref 手动清 |
| 浏览器 | 如 file input 上传后的值等 | 否 |
| 父级 / 外部 | Context、Redux 等 | 视实现而定 |

**关键问**：这次「重置」是**上述各层都清干净**，还是只清了 React？

---

### D. 异步与时序

#### T4b（P0）— 异步与陈旧闭包

**何时用**：`async` 在 **`await` 之后**读取**旧 state** 再 `setState`。

**红旗代码**（示例）：

```javascript
const handleDelete = async (id) => {
  await apiClient.delete(/* ... */); // 等待期间 state 可能已被其它操作改变
  setItems(items.filter((x) => x.id !== id)); // items 可能是闭包快照，已陈旧
};
// 更稳妥：setItems((prev) => prev.filter((x) => x.id !== id))
```

**关键问**：在 `await` 与 `setState` 之间，是否可能有**其它操作**已改变同一份 state？若可能 → 陈旧闭包候选，通常对应高风险。

#### M5（P0）— 请求竞态与过时响应

**何时用**：同一数据源上连续多次请求。

**关键问**：慢请求后返回是否会**覆盖**更新、正确的结果？是否需 **AbortController**、请求序号、或「只采纳最新一次」？

#### M2（P1）— 时序、防抖与副作用顺序

**何时用**：搜索防抖、连点、依赖 API 返回顺序。

**检查要点**：防抖与 **M5** 叠加时，**最终展示**是否正确。

---

### E. 校验与数据变换

#### T4c（P0）— 同一字段多处校验一致性

**何时用**：同一字段在「可提交」「上传」「错误高亮」等多处被判断。

**红旗代码**（示例）：

```javascript
// 两处守卫语义不一致：
isFormValid(): title.trim() !== ''  // 去空白
upload():      !formData.title       // 未 trim
```

**关键问**：对字段 X 的**规范化**（如 `trim`）是否处处一致？不一致 → 至少一处错误；**Risk 2 或 3** 可结合**用户是否总能触达**错误路径（可触达性）判断。

#### T4d（P1）— 纯函数、正则与边界输入

**何时用**：内联 `replace`、正则、`split`、日期/数字解析。

**边界示例**（需在脑中过一遍）：

| 变换 | 正常输入 | 建议考虑的边界 |
|------|----------|----------------|
| `name.replace(/\.[^/.]+$/, '')` | `"report.pdf"` | 无扩展名、`".hidden"`、多段点如 `"a.b.c.pdf"` |
| 日期/数字格式 | `"2024-01-01"` | 空串、`null`、非预期格式 |

**关键问**：每种边界下，**返回值是否仍符合调用方预期**？

#### M1（P1）— 控制模式与展示/实际值一致

**何时用**：表单、mask、只读展示。

**检查要点**：受控/非受控是否混用；**界面显示**与 **`onSubmit`/回调传出值**是否一致（格式化、trim、单位）。

---

### F. React 更新与渲染模型

#### T4f（P0）— 渲染路径副作用

**何时用**：在 **render、`useMemo`、渲染期调用的 helper** 里写 state、ref、外部变量。

**红旗代码**（示例）：

```javascript
const renderRow = (row) => {
  if (matches) {
    expanded.add(row.path); // 在 render 路径中改 Set
  }
  return <Row /* ... */ />;
};
```

React 可能多次调用 render（Strict Mode、Suspense）。此类副作用可能导致状态无限膨胀、无法清理（例如搜索清空后仍无法收起自动展开的行）。

**关键问**：渲染路径中的函数是否**写入** state、ref 或外部变量？→ 非幂等 render，高风险。

#### T4f+（P1）— 重复渲染环境

在 **T4f** 基础上：**Suspense** 与异步子树会使 render 次数与顺序更复杂；「在 render 里改 Map/Set」类写法优先标高风险。

#### T4e（P1）— 非 React 对象原地修改

**何时用**：对 props、context、API 返回对象直接改属性，而非 `setState`/不可变更新。

**红旗代码**（示例）：

```javascript
const handleSave = (row) => {
  row.annotation = editState[row.path]; // 直接改对象
  setEditState(/* ... */); // React 可能因引用未变而不反映 row 的展示
};
```

React 仅在**参与渲染的状态**变化时更新；对普通对象原地赋值可能**不触发**与该行相关的 UI 更新。

**关键问**：是否存在 `x.field = value` 却未通过 **setState / dispatch / 不可变更新** 的路径？

#### M4（P1）— Effect、订阅与定时器

**何时用**：`useEffect` 拉数、订阅、`setInterval`、异步回调。

**检查要点**：依赖是否遗漏导致陈旧值；卸载是否取消请求/订阅/定时器；避免卸载后 **setState**。

---

### G. 列表与协调

#### M6（P1）— 列表 `key` 与行级状态

**何时用**：可编辑行、展开行、行内表单、重排/过滤列表。

**关键问**：是否用 **`index` 作 key** 导致重排后**状态挂错行**？是否应用**稳定 id**？

### H. 元信息

#### M3（P2）— `[SA]` / `[SB]` 与实现类规则

| 标签 | 侧重 |
|------|------|
| **[SB]** | UI/文案/属性**承诺**，以及多路径是否一致、重置是否覆盖各层等**用户可见契约** |
| **[SA]** | 代码**实际行为**；**实现扫描**（陈旧闭包、校验分叉、边界、原地修改、渲染副作用）**多数标 [SA]**，勿强行全部写成 `[SB]` |

**风险**：任意 **[SB]** 规则若代码**未兑现承诺** → 主流程中通常对应 **Risk 3 / P0** 优先处理（与主提示词 **阶段 B** 一致）。

## 四、规则维度核对（写规则阶段用）

- **输入规则**

## 五、风险等级速记（与主提示词阶段 B 一致）

| 等级 | 含义 |

| **3（高）** | 数据错误、错提交、状态不一致、异步问题等；**[SB] 承诺未在代码中落实 → 通常为 3** |

## 七、提交前自检（四技 + 覆盖维度）

**四技（契约扫描）**——任一项为「否」则回到阶段 A 补规则：

- [ ] **T1**：是否找全 UI 承诺（`accept`、`disabled`、状态文案、注释意图等），且**每条在全部路径上**都被代码落实？
- [ ] **T2**：每个「同一用户目标、多路径」的场景，是否验证**共用同一套校验/逻辑**？
- [ ] **T3**：每次 reset/clear，是否审计 **React + DOM + 相关浏览器/外部状态**？
- [ ] **T4**：是否扫描 **异步陈旧闭包（T4b）**、**同一字段多处校验（T4c）**、**内联变换的边界输入（T4d）**？

**质量与覆盖**：

- [ ] 无重复场景；聚焦「**容易真坏**」，而非「理论上可能」
- [ ] 覆盖维度是否点到：**控制方式**、**数据一致性**、**副作用/API**、**时序**（若存在）

## 九、黄金原则

不要追求**穷举**。

而应：**在每条用例上最大化「发现真实缺陷」的概率**。
