# 前端组件测试场景生成流程（防漏测版 v2）

> 目标：在写/审查前端单测时，系统性发现“代码已实现但未覆盖”和“产品期望但代码未实现”的场景，避免漏掉关键边界，尤其覆盖组件在不同状态、控制模式和交互路径下的行为一致性。

---

## 1. 先拆业务规则，再写 case（核心增强）

先从组件代码提炼规则，不要直接写断言。

---

### 1.1 输入规则

- 支持值、非法值、空值、边界值
- **输入中间态（新增）**
  - 输入未完成（如输入一半）
  - 输入过程中是否触发副作用（请求 / 校验）
  - 输入取消（ESC / blur）

---

### 1.2 状态规则

- 初始态、加载态、成功态、失败态、恢复态
- **可用性状态（新增）**
  - disabled 是否完全禁止交互
  - readonly 是否禁止修改但允许查看
- **数据一致性（新增）**
  - UI 显示值 vs 实际提交值（label vs value）

---

### 1.3 控制模式规则（新增重点）

- 受控组件（value 由 props 控制）
- 非受控组件（内部 state）
- 外部更新是否覆盖内部状态
- onChange 是否正确触发
- 是否存在受控/非受控切换问题

---

### 1.4 交互规则

- 点击、键盘、拖拽、多入口一致性
- **焦点与键盘（新增）**
  - focus / blur 行为
  - Tab 顺序
  - 键盘操作（↑ ↓ Enter ESC）
  - 焦点是否正确回收

---

### 1.5 请求规则

- 请求次数、请求参数
- 成功后副作用、失败后回滚
- **触发时机（新增）**
  - onChange vs onBlur
  - debounce / throttle 行为
  - 异步校验触发时机

---

### 1.6 生命周期规则（新增）

- 组件卸载（unmount）是否清理副作用
- 重新挂载（remount）状态是否正确
- 未完成请求是否取消

---

### 1.7 结构规则（针对复杂组件）（新增）

适用于 Tree / Cascader 等：

- 展开 / 收起状态
- 父子节点联动
- 半选状态（indeterminate）

---

### 1.8 原生行为规则（新增）

- 浏览器默认行为是否被正确处理
- change / input 事件触发是否符合预期

---
## 2. 每条规则都必须覆盖完整行为空间

---

### 2.1 正常路径（Happy Path）

- 符合条件时应成功执行
- 行为、状态、数据均符合预期

---

### 2.2 异常路径（Invalid / Error Path）

- 非法输入或不满足条件时：
  - 应被阻止 / 拒绝
  - 或给出明确错误提示
- 不应产生副作用（如错误请求）

---

### 2.3 边界与极端场景（Boundary）

- 空值（null / undefined / 空字符串）
- 极值（最大 / 最小 / 超长）
- 临界状态（刚好满足 / 刚好不满足）

---

### 2.4 行为扰动场景（Interaction Stress）

模拟真实用户“非理想操作”：

- **重复操作**
  - 重复点击
  - 重复输入相同值
- **中断操作**
  - 输入未完成即 blur / ESC
  - 请求过程中被取消 / 覆盖
- **快速连续操作**
  - 多次快速触发（可能引发竞态）
- **状态切换中操作**
  - loading → 再次触发操作

---

### 统一原则

每条规则至少要回答：

> 在「正常 / 异常 / 边界 / 扰动」四种情况下，组件行为是否仍然正确且一致？
---

## 3. 明确三类测试目标（必须分开）

- `行为验证`：验证当前实现确实这么做
- `规范验证`：验证产品期望应该这么做
- `缺陷固化`：已知 bug 用 `it.fails` / `it.todo` 固化

---

## 4. 多入口一致性检查（统一规则）

若同一能力存在多个入口（如 input / drag、鼠标 / 键盘、UI 操作 / 外部 props）：

必须保证在不同入口下：

- UI 行为一致
- 数据结果一致
- 触发时机一致

> 即：不同入口只是触发方式不同，不应导致行为、结果或时机的差异

---

## 5. 真实用户场景优先级排序

- P0（阻断）：功能不可用、错误数据提交、状态错乱
- P1（高频）：用户常用路径、重复操作、网络波动
- P2（体验）：提示文案、视觉反馈、极端输入

---

### P0 必须包含（新增）：

- 控制模式错误（受控失效）
- 值不一致（显示 vs 提交）
- disabled 仍可操作
- 请求时机错误

---

## 6. 自动化层级判定（Unit / E2E / 手工）

---

### Unit

- 状态转换
- 参数计算
- 控制模式逻辑
- debounce / throttle

---

### E2E

- 键盘操作
- focus / blur
- 浏览器行为（input / upload）
- drag / drop

---

### 手工

- 视觉表现
- 浏览器差异
- 原生行为差异

---

## 7. 变更触发器清单（统一规则）

当代码或需求中出现以下关键词时，必须进行专项复查：

- 数据与提交：upload / submit / delete / confirm / fetch
- 输入与文件：FormData / accept / drag / drop
- 状态更新：setState（链式更新 / 闭包旧值）
- 控制模式：value / defaultValue
- 可用性控制：disabled / readonly
- 时序控制：debounce / throttle
- 生命周期：key / 条件渲染（组件重建）
- 事件触发：onChange / onBlur

> 即：凡涉及“数据提交、状态更新、控制方式、时序或生命周期”的变更，必须重点检查是否引入漏测或行为变化

---

## 8. 常见漏测点速查（统一清单）

在审查测试用例时，需重点检查是否遗漏以下场景：

- 只测成功，未覆盖失败与恢复流程
- 只测支持输入，未覆盖非法输入或异常输入
- 只测单一入口，未覆盖多入口（点击 / 拖拽 / 键盘）
- 只测单次操作，未覆盖重复或连续操作
- 只测请求触发，未验证请求后的 UI 副作用

- 受控组件未验证外部更新是否生效
- 输入中间态（未完成输入）未覆盖
- 焦点管理或键盘操作缺失（Tab / Enter / ESC 等）
- disabled / readonly 状态下仍可交互
- 显示值与提交值不一致（label vs value）
- debounce / throttle 导致行为或时机异常
- 组件卸载后副作用未清理（如请求 / 事件）
- 层级组件（如 Tree）父子状态不一致

> 若以上任一项未覆盖，需补充对应测试用例

---

## 9. 已知缺陷固化规范（it.fails）

- **命名规则**：每个已知缺陷一个独立 `it.fails`（标题中带缺陷编号/关键字即可，避免歧义）
- **断言口径**：`it.fails` 断言的是**产品期望**（Should），不是当前实现（As-is）
- **稳定性**：若出现 `Expect test to fail`，通常说明该用例在当前环境下“意外通过”（可能被测试工具行为掩盖），需要调整触发方式使其稳定暴露缺陷
- **生命周期**：缺陷修复后必须将 `it.fails` 改为普通 `it`（否则会阻止你发现回归）

---

## 10. 文件上传组件专项清单（保留）

适用于包含 `<input type="file">` 或拖拽上传的组件（抽取规则）：

- **双入口一致性**：input 选择与 drag/drop 必须遵循同一校验与提示规则（正向 + 反向都要测）
- **连续操作序列**：至少覆盖“上传 -> 删除 -> 再上传”（必要时覆盖“同一文件重传”）
- **请求副作用**：成功后刷新/更新列表；失败后提示且可恢复再次提交
- **控件细节**：关注“同文件重复选择不触发 change”等浏览器行为差异（必要时以 E2E/手工补齐）

---

## 11. 错误展示规则（新增）

- 错误提示是否出现
- 是否可关闭 / 自动消失
- 多次错误是否叠加
- 是否影响后续操作

---

## 12. 交付物要求（每次迭代）

- 覆盖矩阵：规则 -> case 映射
- 已知缺陷清单：`it.fails`
- 手动验证清单（偏"检查点格式"，用于人工核对）
- Unit / Manual 对比矩阵（用于判定回归补测缺口）

---

## 12.1 手动验证清单（检查点格式）

目标：让人工执行可以逐条勾选，并且每条都能映射到对应的 unit case（或明确说明为什么 unit 不易覆盖）。

### 输出建议（可读性优先）

- **默认输出为“表格版检查点”**：先把所有手动项用一张表列清楚（类似 `DocumentUpload-test-analysis.md` 的 Manual 表格），每行就是一个“可勾选的检查点”。
- **仅对少数复杂/高风险项展开为 checkbox 细化清单**：例如竞态、原生行为差异（文件选择器/拖拽 accept）、键盘/焦点、视觉提示、刷新后持久化等。

> 原则：不改变“检查点约束”（仍需可勾选、可回归、可对齐 unit），只是**换一种更易读的呈现方式**。

### 表格版检查点（推荐默认格式）

| 场景ID | 中文动作名 | 优先级(P0/P1/P2) | 入口对象 | 前置条件(P)摘要 | 操作(A)摘要 | 期望结果(E)检查点摘要 | 证据口径（Unit/Manual） | 备注 |
| ------ | ---------- | ----------------- | -------- | --------------- | ----------- | ---------------------- | ------------------------ | ---- |

每一行都必须满足：
- 一个 `场景ID` 对应一个中文动作（原子动作）
- `期望结果(E)` 只写该动作的产物（UI/数据一致性/副作用其一或组合）
- 明确 `证据口径`：`Unit 可断言` / `需要 Manual 才稳定`

---

当某条表格检查点需要拆更细（例如竞态/恢复/原生差异），**不要改用长模板**，而是把“一个大检查点”拆成**多行表格检查点**：

- 同一个“动作”可拆成多个 `场景ID`（例如 `AB-M-03a/03b/03c`），每行只写一个可勾选产物
- 在 `期望结果(E)检查点摘要` 中用分号列出 2–3 个最关键检查点（仍然只属于同一个动作产物）

---

## 12.2 Unit / Manual 对比矩阵

用于每次迭代输出“还差哪些回归点”，避免 manual 和 unit 脱节。

| 场景ID | 中文动作名 | 优先级(P0/P1/P2) | 规则来源 | Unit是否覆盖 | Unit断言产物（简述） | Manual是否需要 | Manual检查点摘要 | 备注（差异/原因） |
| -------- | ----------- | ----------------- | -------- | ------------ | ---------------------- | -------------- | ------------------- | ------------------ |

判定规则（建议写进模板里强制执行）：
- 如果存在 `SB(UI 承诺)` 且 unit 难以稳定验证（焦点/键盘/原生文件输入/拖拽 accept/视觉提示），则 `Manual是否需要=是`。
- 如果场景已用 `it.fails` 固化为缺陷哨兵，则 `Manual是否需要=是`。
- 若 `Unit是否覆盖=是`，`Manual是否需要` 默认可为 `否`；仅在差异风险（browser/原生/时序/恢复）较高时才改为 `是`。

---

## 12.3 简化输出模式（推荐默认）

> 目标：让输出文档更像“测试分析报告”（易读、可对账、可回归），而不是“长模板堆叠”。

当你希望输出更简洁时（推荐作为默认模式），按以下顺序组织交付物：

1) **Step 0 摘要（1 页内）**：列出最关键的 UI Promise / 多入口一致性 / 状态层审计 / 实现漏洞（只保留 Risk=3/2）
2) **规则与风险表**：`规则ID | SA/SB | 风险(P0/P1/P2) | 一句话说明 | 覆盖状态(Unit/Manual/缺口)`
3) **已知缺陷（it.fails）小表**：`Bug | 期望(Should) | 现状(As-is) | 哨兵用例名`
4) **Unit / Manual 对比矩阵（12.2）**：标缺口、标原因
5) **Manual 表格版检查点**（12.1 表格）：按 P0/P1/P2 排序
6) **仅对少数条目展开 checkbox 细化清单**：用于复杂/高风险项

> 注意：简化模式不减少内容要求，只减少“展开文字”；所有细节仍可通过表格字段表达并可回溯。

## 13. 执行模板（升级版）

1) 阅读组件，列规则（含控制模式）  
2) 生成正反向场景表  
3) 标记 Unit / E2E / 手工  
4) 优先覆盖 P0（控制模式 / 一致性 / 时机）  
5) 加入输入中间态 & 生命周期  
6) 对已知 bug 加 `it.fails`  
7) 输出覆盖矩阵  
8) 生成手动验证清单（检查点格式）  
9) 输出 Unit / Manual 对比矩阵并标注缺口  
10) 回归执行  

---

# 一句话原则（升级版）

> 不仅要验证“功能是否正确”，  
> 还要验证“在不同控制方式、交互路径、输入阶段和生命周期下，行为始终正确”。
